文章接口文档

获取文章列表

无论是发现模块还是我的模块，这两部分中都涉及到文章列表的获取，因此可以考虑将其设计成一个单独的接口，而非给每个模块单独设计接口。

前端请求

  • 请求地址：
  • 请求方式：POST
  • 请求参数：
dhcjs: 1                      // 检索类型， 0：无检索类型；1：专题号；2：用户账号
bvnsj: "123"                  // 检索内容
utysh: 1                      // 文章类型，1：最新文章；2：最热门文章
idonx: "2018-03-05 18:18"/10  // 划分条件

【说明】：

  • 当 dhcjs 的值为 0 时：表示无须检索条件。例如，发现模块中推荐Tab下的文章获取。此时无须根据专题号、用户账号来进行筛选。
  • 当 utysh 的值为 1 时：idonx 表示的是已获取文章中的最早日期。此时 idonx 的值类型为日期（例如 "2018-03-05 18:18"）。
  • 当 utysh 的值为 2 时：idonx 表示的是已获取的文章数。此时 idonx 的值类型为数字（例如 10）。

【示例】：

// 示例一
dhcjs: 0
bvnsj: ""
utysh: 1
idonx: "2018-03-05 18:18"

// 示例二
dhcjs: 0
bvnsj: ""
utysh: 2
idonx: 10

// 示例三
dhcjs: 1
bvnsj: "123"
utysh: 1
idonx: "2018-03-05 18:18"

// 示例四
dhcjs: 1
bvnsj: "123"
utysh: 2
idonx: 10

// 示例五
dhcjs: 2
bvnsj: "879646529"
utysh: 1
idonx: "2018-03-05 18:18"

// 示例六
dhcjs: 2
bvnsj: "879646529"
utysh: 2
idonx: 10

【说明】：

  • view_article 视图已经按照文章日期降序排列（从大到小）；
  • 检索类型表示后台检索文章的方式。举例说明，专题号表示 WHERE sid = dhcjs;用户账号表示 WHERE account = dhcjs;
  • 文章类型表示获取文章列表的排序方式，1 表示最新文章，也就是说文章按照文章日期降序排列（默认），此时需要通过已获取文章中的最早日期来获取之前的文章，所以前端传递的是一个日期。2 表示最热门文章，换言之文章按照阅读数降序排列，此时根据已获取文章数来获取后续的文章（其实这么做也存在一定的问题，比如在获取后续文章的时候，已获取文章和未获取文章之间的排序发生变更，从而导致重复获取已有文章，但暂时不管这种情况）。

获取文章排行榜

用户点击文章排行榜按钮即可进入到文章排行榜界面，此时应用会自动向后台发起请求以获取阅读数最多的文章。

前端请求

  • 请求地址：
  • 请求方式：POST
  • 请求参数：
ieudh: 0        // 已获取的文章数
hfeyd: 1        // 排行榜类型

排行榜类型：

  • 1：24H热门，获取系统时间减去一天。
  • 2：7日热门，获取系统时间减去七天。
  • 3：30日热门，获取系统时间减去三十天。

评论文章

用户可以自由评论文章，也可以对已有的评论进行评论。若对文章评论，则默认@作者；对已有的评论进行评论，则@评论的用户。

前端请求

  • 请求地址：
  • 请求方式：POST
  • 请求参数：
iwuus: "879646528180323135935"  // 文章号
kalcb: "879646529"              // 用户账号
thske: "87964528"               // 被评论者账号
ywhkd: "人生若字如初见"         // 评论内容
mnsjf: 0                        // 评论类型 0：对文章评论，1：对评论进行评论

后台响应

  • 对应数据表：dmesscomment
  • 响应内容：
// 成功响应
{
    "code": "1",
    "msg": "",
    "data": []
}

// 失败响应
{
    "code": "0",
    "msg": "具体原因",
    "data": []
}

【说明】：

  • 后台将前端传递的文章号、用户账号、被评论者账号、评论内容、评论类型依次插入到 dmesscomment 数据表的 Particle、Vsender、Dreceiver、Scontent、Rtype 字段中。Ecid 为自增长无需关注，Odate 字段取当前系统时间，格式为 YYYY-MM-dd HH:mm。

记录文章阅读信息

用户查阅文章后，会将该文章记录到自己的阅读历史中，同时也将增加这篇文章的阅读数。现在的问题是自己阅读自己的文章是否会增加阅读数。

前端请求

  • 请求地址：
  • 请求方式：POST
  • 请求参数：
rqgcm: "879646528180323135935"  // 文章号
ivhws: "879646529"              // 用户账号

后台响应

  • 对应数据表：larticle
  • 响应内容：
// 成功响应
{
    "code": "1",
    "msg": "",
    "data": []
}

// 失败响应
{
    "code": "0",
    "msg": "具体原因",
    "data": []
}

【说明】：

  • 后台根据前端传递的文章号，从 larticle 数据表中找到相应数据，将其 Oread 字段值 +1。
  • 然后再将文章号和用户账号分别插入到 ohistory 数据表的 Gaid 和 Aaccount 字段。Chid 字段自增长无需关注，Pdate 取当前系统时间，格式为 YYYY-MM-dd HH:mm。
  • 注意，因为该接口涉及多项操作，因此需要进行事务处理。

前端请求

  • 请求地址：
  • 请求方式：POST
  • 请求参数：

rwhcs: "879646528180323135935"  // 文章号
uwhgc: "879646529"              // 用户账号

后台响应

  • 对应数据表：fuserlike，hmesslike
  • 响应内容：
// 成功响应
{
    "code": "1",
    "msg": "XXX操作成功，具体说明到时候再议",
    "data": []
}

// 失败响应
{
    "code": "0",
    "msg": "具体原因",
    "data": []
}

【说明】：

  • 后台依次将前端传递的 ivwga、rwhcs、uwhgc 插入到 fuserlike 数据表的 Rauttaccount、Taid、Vaccount 字段中。Hulid 为自增长无需关注，Idate 取当前系统时间（需要进行处理，格式为 YYYY-MM-dd HH:mm）。
  • 修改 larticle 数据表中当前文章号的 Klike 字段，将其值 +1。
  • 插入到 fuserlike 数据表成功后，向作者发送消息，表示他的文章被某某用户喜欢。将 ivwga、rwhcs、uwhgc 分别插入到 hmesslike 数据表的 Ireceiver、Varticle、Ssender 字段中。Tlid 为自增长无需关注，Wdate 取当前系统时间（需要进行处理，格式为 YYYY-MM-dd HH:mm）。
  • 注意，因为该接口涉及多项操作，因此需要进行事务处理。

写作模块

用户可以纂写文章、以及编写文章过程中的图片上上传。

【流程说明】：
首先需要明确的一点是文章和图片是分开上传。问题的关键在于图片上传的时机。若是在最终提交文章时再上传图片，这会让用户多等待一段时间。若在用户插入图片时就上传图片，一旦用户最终放弃编写文章，那么上传的图片就会占据服务器的磁盘空间，造成不必要的浪费。目前想到的比较好的处理方式是：

  1. 文章号不要自增长，通过“用户账号+时间”的格式生成。
  2. 用户进入到写作页面时，前端就会生成一个文章号。在这之后上传的图片都对应这个文章号。这么做的好处在于：用户一旦放弃编辑，前端将这个文章号发送到后台，后台将刚才用户上传的对应该文章号的图片全部从磁盘中删除。
  3. 用户不放弃编辑，最终提交文章，则将前端传递的所有信息都插入到 larticle 数据表中。

图片上传

用户可以为文章添加图片。

前端请求

  • 请求地址：
  • 请求方式：POST
  • 请求参数：
twhck: "879646528180323135935"  // 文章号
oxjwg: "..."                    // base64 图片内容

后台响应

  • 对应数据表：无
  • 响应内容：
// 成功响应
{
    "code": "1",
    "msg": "XXX操作成功，具体说明到时候再议",
    "data": "kaiyan/article/879646528180323135935/180423181312.jpg"
}

// 失败响应
{
    "code": "0",
    "msg": "具体原因",
    "data": []
}

【说明】：后台收到前端传递的图片和文章号时，先到磁盘的指定位置处（暂定为 kaiyan/article），判断是否存在名为文章号的文件夹。若存在，则无需创建文件夹；若不存在，则创建与文章号同名的文件夹。然后再把图片保存到该文件夹下，难点在于图片的重命名。比如说前端上传第一张图片，后台自然是将这张图片命名为 1.jpg，但是之后的图片该如何命名？后台如何知道这张图片是第几张？有两种处理方式，第一种是前端传递图片序号；第二种是以时间作为图片名称。我建议是第二种方式较好（有其他想法可以随时和我说）。举例，取名为 180423181312.jpg（YYMMddHHmmss的格式），因此后台返回 kaiyan/article/879646528180323135935/180423181312.jpg。
